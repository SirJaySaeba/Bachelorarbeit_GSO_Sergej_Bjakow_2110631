Im dem Michael Hartl Tutorial (Kapitel 6) wird ja darauf hingewiesen, dass selbst, wenn ich im model durch "uniequeness: true" Einzigartigkeit gewährleisten möchte, dass dadurch nicht unbedingt garantiert wird.

Durch Doppelklick auf den Submit-Buttons eines Registrierungsformulars soll es passieren können, dass zwei Requests verschickt werden und somit zwei User mit demselben Namen/email in der Datenbank gespeichert werden.

Also ich habe versucht das mal zu provozieren, doch ich erhalte dann nur die flash message für eine erfolgreiche Registrierung (1. request) und, dass die email schon vergeben sei (2.request).

Wie üblich ist diese vermeintliche Sicherheitslücke in der Praxis nun eigentlich und trefft ihr bei euren Rails-Projekten entsprechende Vorkehrungen um dieses sogenannte "uniqueness caveat" vorzubeugen? In dem Tutorial wird der user tabelle dafür extra ein index vergeben...

Das Ereignis kannst du nur sehen, wenn die einen Multithreaded Server einsetzt (Webrick geht da zum testen nicht).

Wie wahrscheinlich das ganze ist hängt von deiner App ab und wird man dir kaum pauschal beantworten können. Hast du zb. viele Validation Regeln beim Create,
dann kann es schnell passieren, dass die Race Condition auftritt, oder das save läuft innerhalb einer lang laufend Transaktion. Die Frage ist auch, wie wahrschleinlich ist es das entweder dein JS zu langsam ist und das Formular 2 x abgeschickt wird und wie Wahrschlich ist es, dass ein 2. User einen Datensatz mit den gleichen Daten anlegt.

Damit kannst du das auch mal simulieren, bau einfach eine 2. Validation ein, die nix macht als 5 sek. sleep
und dann true liefert. Füge diese Validation Regel nach der uniq: true ein, dann solltest du die Race Condition leicht erzeugen können, oder Schachtle das save in eine Transaktion und mach dort ein sleep. Dann solltest du theoretisch das Ergebnis sehen, wenn du einen Server wie unicorn verwendest.

Abfangen kann man das DB seitig ganz einfach mit einem Uniq-Index, wenn's wirklich nötig sein sollte.

Ganz klar ja. Das ist eine klassische Race-Condition und die Häufigkeit des Auftretens hängt stark davon ab, wie viel Traffic deine Seite hat. Sich dagegen zu schützen geht mit einem Index auf der Datenbank recht einfach - das Problem rauszupuzzeln wenn's denn mal vorgekommen ist ist ganz schön viel Arbeit. Ganz klarer Fall von "wer vorher dran denkt spart sich viel Arbeit."
